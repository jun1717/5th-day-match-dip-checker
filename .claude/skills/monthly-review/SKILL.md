# 月次振り返り

[docs/運用マニュアル.md](../../../docs/運用マニュアル.md) のセクション4「月次」を実行・解釈する手順。
目的は「勝てているか」ではなく「**執行の問題かルールの問題かを切り分ける**」こと。1ヶ月の損益で一喜一憂する場ではない。

## 手順

対象月: 引数の `YYYY-MM`。省略時は実行日の前月。

1. **同期と価格更新**

   ```bash
   git pull
   python3 scripts/fetch_prices.py
   ```

   - `git pull` はGitHub Actionsが毎日16:30 JSTにコミットするシグナル履歴(`data/history/`)の取得。失敗したら(オフライン等)その旨を報告して続行する。
   - 対象月が1年以上前の場合のみ `--prices data/prices_backtest.csv` を使う(事前に `--period 2y` で取得)。

2. **蓄積状況の確認**: `data/history/signals/` のファイル数と最古・最新の日付を確認する。対象月の営業日ぶんのスナップショットが揃っていなければ、欠落日を報告する(review:tradesの突き合わせ精度に影響する)。

3. **実売買レビュー**

   ```bash
   npm run review:trades -- --month <YYYY-MM>
   ```

   `data/trades/executions.csv` がヘッダーのみ(記録なし)の場合は「実売買記録がないため執行レビューはスキップ。約定当日の記録を忘れていないか」と報告し、手順5へ。

4. **review:tradesの解釈**(この優先順位を必ず守る)

   1. **`holding_below_stop`(損切りライン割れを保有中)** — 最優先。該当建玉を銘柄・現在値・損切りラインつきで列挙し、レポートの冒頭に警告として出す。「翌朝どうするか今決める」ことを促す。
   2. **逸脱フラグ集計** — `not_buy_candidate`(ルール外エントリー)・`chase_entry`(買い上限超え)・`late_stop`(損切り遅れ)の件数。目標はゼロ。`same_day_signal`(場中判断)は別枠(執行タイミングの問題)として数える。
   3. **実成績 vs 仮想成績の差** — 差は純粋に執行の質(仮想成績は実際の株数で計算される)。
      - 実成績 < 仮想成績 → **執行の問題**。ルールは触らず、逆指値の事前セット等の執行改善を具体的に提案する。
      - 仮想成績そのものが悪い → **ルールの問題の可能性**。ただし単月では判断せず、複数月の傾向として溜まったら `/backtest-ab` での検証を提案する。

5. **シグナル履歴分析**

   ```bash
   npm run analyze:signals
   ```

   - status別(buy_candidate / watch)のフォワードリターンと約定シミュレーション成績を要約する。
   - **履歴は2026年7月から蓄積開始**。2〜3ヶ月分未満の間は「蓄積中(n=◯日分)。判断には使わない」と明記し、数字の深読みをしない。
   - 参照基準は2年バックテスト: **期待値R 0.619 / PF 1.60 / 勝率22.1%**([docs/backtest-results/2026-07-atr-volume.md](../../../docs/backtest-results/2026-07-atr-volume.md))。勝率2割強で期待値を出す設計なので、数連敗は正常の範囲であることを毎回明記する。

## レポート形式

チャットに以下の順で簡潔に出す(ファイルは作らない。CSVは `data/analysis/` に自動保存済み):

1. ⚠️ 損切り割れ保有(あれば冒頭)
2. 対象月のトレード数・実損益・仮想損益・差額
3. 逸脱フラグ集計(ゼロなら「規律OK」と一言)
4. 結論: 執行の問題/ルールの問題/どちらでもない、のどれか一言 + 来月に向けた具体的アクション(あれば1〜2個)
5. シグナル履歴の蓄積状況とstatus別サマリ

## してはいけないこと

- このスキル内で `config/rules.json` を変更しない。ルール変更の入口は `/backtest-ab` のみ。
- 単月の結果から閾値変更を提案しない(検証済み既定値の一覧は運用マニュアルのセクション6)。
- `executions.csv` への行追加をユーザーの代わりに推測で行わない(約定記録はユーザーにしか分からない)。
